Server and Content Migrations

The following are types of content or server migrations I have been apart of.

DTD Schema Migrations

  • Moved several XML manuals from the Aircraft Maintenance Manual (AMM) DTD schema to a the Flightbook schema, which is more industry accepted
  • Required line by line copy and paste from old XML schema to new XML schema
  • Approximately 3,000 pages of content moved
  • Rigorous quality reviews

CMS Content Migration - Adobe FrameMaker to Arbortext

  • Moved several manuals from Adobe FrameMaker into Arbortext.
  • Required line by line copy and paste from PDF to XML
  • Approximately 2,000 pages of content moved

CMS Software Upgrades

  • Upgraded TechSuite multiple times over the years.
  • Each upgrade required detailed quality checks of all functions, in Dev, QA, and Production environments.
  • Worked with vendors, various technology groups, and coordinated with technical writers to ensure minimal work disruption.

Brand New CMS Software Install

  • Oracle content database were dated and no longer going to be supported. The new Oracle database demanded that TechSuite (Arbortext) also be updated at the same time.
  • Initially, Technology implemented a plan tp just upgrade TechSuite and the database to keep costs low. But FlatIrons (TechSuite Vendor) did not have proper documentation and created multiple upgrade issues.
  • The upgrade ran over the alloted time and project budget for several months with little to no progress. We decided on a solution of a clean new database and TechSuite install. This required manually moving all content (100s of manuals and documents).
  • I led a team of 5 people to migrate the content from the old database to the new. We keet a copy of 100% of data, but only uploaded the last two revisions and any drafted content into the new content management system.
  • Worked with cross-function teams to stand new database, upgrade TechSuite, and coordinate content movement.

Brand New Web Server

  • Southwest Airlines Employee website (SWALife) moved from IBM WebSphere + Documentum to Microsoft SharePoint, which does not support HTML manuals. A new content web server was created for all manuals (both PDF and HTML), and all manuals are now located on this web server.
  • Worked with multiple groups to stand up new web server, set up security groups and SSO, work around revision schedules, find all links to our manuals in Documentum, and update links within manuals to point to new URL.
  • Created forwarding/redirect pages for PDF and HTML. Could not just have redirect links since the entire Documentum webserver was shut down.
  • Created automation to update all links in all manuals to point to new URL. Each manual can be completed in under 1 minute, saving time and reducing risk. Refer to Web Server Hyperlink Automation.
  • Worked with Technology groups to set up a Tivoli script to push manuals to the Central Pubs Web Server during the next available synchronization time, which occurs daily at 7:00 AM, 9:00 AM, 11:00 AM, 1:00 PM, 3:00 PM, 5:00 PM, and 1:00 AM Central Standard Time.
  • Created new documentation for how to publish content on the new web server, maintaining the werbserver (e.g., restarts, scripts, contacts), naming conventions, and FAQ help.
